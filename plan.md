# NamazuHaUrokoGaNai — 自宅地震計システム 実装計画

## Context

Qiita記事「地震観測用加速度センサ5種の性能比較」( https://qiita.com/compo031/items/e62d0a0e1425c5e1efe8 ) に触発され、購入済みの IIS3DHHC 加速度センサ + ESP32 で 24時間365日稼働の自宅地震計を作る。測定 → 送信 → 保存 → 検知/通知 → 可視化 の全パイプラインを構築する。

記事の要点: 100Hzサンプリング、気象庁の計測震度算出法（周波数フィルタ + 3成分合成の降順30位 → `I = 2×log10(a) + 0.94`）。IIS3DHHCはノイズフロアが計測震度0.0相当で有感基準(0.5)以下、個人地震計に十分。

### ユーザーの決定事項

- マイコン: 手持ちの **ESP32-D0WDQ6**（無印デュアルコア、おそらくWROOM-32系DevKit）。必要ならS3購入も可
- 検知方式: **ハイブリッド**（デバイス側でリアルタイム検知して即通知 + クラウド側で保存・検証・イベント切り出し）
- IaC: **Terraform**
- 通知先: **Slack**（Incoming Webhook）。ただし差し替え可能な口にする

### 設計上のツッコミ（planに反映済み）

1. **IIS3DHHCのODRは1.1kHz固定**。100Hzには設定できない。10ms周期タイマーで最新値をポーリングする（記事と同じ方式）。センサ内蔵LPF＋震度フィルタが高周波を殺すので実用上問題なし
2. **コストの支配項はLambdaでなくS3 PUT**。生データは600B/s程度で、30秒バッチなら86k PUT/月 ≈ $0.43。Lambdaは無料枠内。送信間隔を1秒にするとPUTだけで月$13になる → ハイブリッド方式なら速報性をデバイス側が担保するので**バッチは30秒でよい**
3. **バッチ境界を跨ぐ揺れ問題**は、気象庁の正式算出法（60秒窓FFT）を使うから起きる。**リアルタイム震度**（功刀ら2013の時間領域IIRフィルタ近似。強震モニタで採用）を使えばフィルタ状態を持ち回れて境界問題が消える。しかも100Hz設計のフィルタなのでESP32上でそのまま動く
4. 24/365稼働にはWiFi断/AWS障害時の**ローカルバッファリング**（LittleFS退避、復旧後バックフィル）とwatchdogが必須
5. S3ライフサイクルは「prefix単位のexpiration」しかできない。「検知イベント周辺だけ長期保存」は**検知時に `events/` prefixへコピー**することで実現する（`raw/` は90日でexpire）

## アーキテクチャ

```
[IIS3DHHC] --SPI--> [ESP32]
                      ├─ 測定タスク(Core1): 100Hzポーリング → リングバッファ
                      ├─ 検知タスク: リアルタイム震度IIR常時計算 → 閾値超え→即時アラートPOST
                      └─ 送信タスク(Core0): 30秒バッチPOST / NTP / LittleFS退避・バックフィル
                            │ HTTPS (Lambda Function URL, HMAC認証)
                            v
[ingest Lambda] ─→ S3 raw/YYYY/MM/DD/HH/<batch>.bin  (lifecycle: 90日でexpire)
       │ (非同期起動)
       v
[detect Lambda] ─ 直近~2分をS3から読み、リアルタイム震度を再計算・検証
       ├─→ DynamoDB events テーブル（イベント記録・アラート重複排除）
       ├─→ S3 events/<event-id>/ へ波形コピー（永久保存）
       └─→ Slack Webhook 通知（通知層はinterface化して差し替え可能に）

[dashboard] S3+CloudFront 静的SPA ←→ [api Lambda] 直近n分波形・イベント一覧を返す
```

### データ形式

- バッチはコンパクトなバイナリ: ヘッダ（magic/version, device_id, batch開始時刻µs, サンプルレート, サンプル数, **スケール係数(LSB→mg)**, センサ種別）+ int16×3軸×N
- スケール係数とセンサ種別をヘッダに持たせることで、**将来ADXL355（20bit）に差し替えてもパイプラインが壊れない**（その際はint32格納に拡張）
- ファームウェアのセンサ読み出しは `AccelSensor` インターフェイス（init/read/scale）に抽象化し、IIS3DHHC実装をその背後に置く

### リアルタイム震度（両側で同一アルゴリズム）

- 功刀ら(2013)の近似フィルタ（バイキュアッド直列のIIR、100Hzサンプリング用に公開された係数を使用。既存OSS実装を参照）
- フィルタ後3成分合成加速度の直近60秒リングバッファから「累計0.3秒（=30サンプル）超過値」aを取り `I = 2log10(a) + 0.94`
- **生活振動の除去は継続時間ベース**: 扉の開閉・物の落下等は一瞬（<0.5秒）で終わるという仮説に基づく。「累計0.3秒超過値」という定義自体が単発スパイクを抑制する上、検知条件を「リアルタイム震度 ≥ 0.5 が数秒継続」とすることで、数秒以上続く地震と分離する
- 閾値・継続時間のチューニングは、**生データが90日残るのでクラウド側でバックテスト可能**にする（detect Lambdaのコアロジックはローカル実行可能なpure関数として書く）

### 送信の信頼性（リトライ・バックフィル）

- 送信タスクは「**2xx ackが返るまでバッチを未送信キューから消さない**」の一本のルールで動く。失敗理由（WiFi断/AWS障害/一時的HTTPエラー）は区別不要
- 失敗時は指数バックオフでリトライ。RAMキューが溢れそうならLittleFSへ退避し、復旧後に古い順でバックフィル
- S3オブジェクトキーは受信時刻でなく**測定開始時刻**から決める → バックフィルされたバッチは正しい時間位置に格納され、ダッシュボードは時刻範囲でS3を引くだけなので**欠測穴が後から自然に埋まる**。二重送信も同一キーへの上書きになり冪等

### 時刻同期

- esp_sntpで起動時+定期同期（step でなく slew 調整、測定中に時刻が飛ばないように）
- バッチには開始時刻+サンプルindexでタイムスタンプ付与。ingest側で受信時刻も記録（ドリフト監視用）

## リポジトリ構成

```
firmware/    PlatformIO (Arduino core for ESP32; watchdog/SNTP/OTA等はIDF APIを直接呼ぶ)
lambda/      Python (ingest / detect / api)。scipyでフィルタ、バックテストしやすい
terraform/   S3, Lambda×3, Function URL, DynamoDB, lifecycle, IAM
dashboard/   静的SPA (TypeScript + 軽量チャート)
tools/       シリアルキャプチャ・震度計算・バックテスト用Pythonスクリプト
docs/        設計メモ（本planの内容を反映）
```

## 実装フェーズ

### Phase 1: センサ検証（クラウドなし）
- PlatformIOプロジェクト作成、IIS3DHHC SPIドライバ（ライブラリ不使用、記事同様レジスタ直叩き）
- 10msタイマーで100Hzサンプリング → シリアル出力
- `tools/` に記事の方式（FFT+気象庁フィルタ）で計測震度を算出するPythonスクリプトを作り、**ノイズフロアが記事の「計測震度0.0」相当か検証**
- リアルタイム震度IIRフィルタもここでPython実装し、FFT版と結果照合

### Phase 2: クラウドingest
- Terraform: S3バケット（`raw/` 90日lifecycle）、ingest Lambda + Function URL（HMACヘッダ認証）
- ファームウェア: 測定/送信のタスク分離、30秒バッチのバイナリPOST、NTP同期、WiFi再接続、LittleFS退避+バックフィル、watchdog
- ingest Lambda: 認証検証 → S3へ格納

### Phase 3: 検知・通知（ハイブリッド）
- ESP32: リアルタイム震度の常時計算、閾値超えで即時アラートPOST
- detect Lambda: バッチ到着ごとに直近~2分で震度再計算・イベント判定、DynamoDBで重複排除、`events/` へ波形コピー、Slack通知（Notifierをinterface化）
- デバイスアラートとクラウド検知は同一イベントテーブルで突合（デバイス発が速報、クラウド発が確定報）

### Phase 4: ダッシュボード
- api Lambda: 直近n分（n=1,3,5,10,60）の波形をデシメーションして返す + イベント一覧/イベント波形
- 静的SPA: 現在の揺れグラフ（数秒ポーリング）、イベントブラウザ
- S3+CloudFront配信（認証なし）

### Phase 5: 運用強化（後回し可）
- OTA更新、デバイスのヘルスメトリクス（欠測監視 → 「データが来ない」こと自体のアラート）
- 生活振動除去のバックテストとチューニング
- （気が向いたら）ADXL355追加

## コスト見込み

S3ストレージ ~5GB（90日分）+ PUT 86k/月 + Lambda無料枠内 + DynamoDBオンデマンド ≈ **月$1未満**

## 検証方法

- Phase 1: 静置ノイズの計測震度が0近傍であること。机を叩く/ジャンプで応答確認
- Phase 2: S3にバッチが並ぶこと。WiFiを切って復旧 → 欠測なくバックフィルされること
- Phase 3: 加振でSlack通知が来ること（デバイス速報→クラウド確定報の順）。`events/` に波形が残ること
- Phase 4: ダッシュボードで直近波形とイベント波形が見えること
- 長期: 数日連続稼働で欠測率とNTPドリフトを確認
